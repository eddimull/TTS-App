## Problem

Amending a contract that is out for signature requires a manual, error-prone ritual:
change the booking status from pending back to draft, log into PandaDoc, delete the
document, edit the terms, and resend. Neither frontend offers any affordance â€” web
shows "This contract is pending. The contract is no longer editable." and mobile
hard-locks `ContractDefaultView` into a read-only preview.

Separately, the mobile API's `UpdateBookingRequest` silently drops `contract_option`
(not in its rules), which strands `none` bookings ("Change to a contract type" is a
"coming soon" stub) and blocks post-create type changes that web already allows.

## Decisions (agreed 2026-07-05)

- Ship backend + **both** frontends in this effort.
- **Void** the PandaDoc document (`PATCH /public/v1/documents/{id}/status/`, voided
  code) rather than delete â€” recipient's link shows a voided document, PandaDoc-side
  audit survives. A PandaDoc **404 counts as success** (doc already deleted manually)
  so amend is idempotent.
- **Unsigned only**: amend is available while contract status is `sent` and booking
  status is `pending`. Signed/completed contracts stay locked forever.
- Include the `contract_option` wiring (mobile validation rule + functional picker).

## Backend (TTS)

### Recall / void

New `voidPandaDocDocument()` next to the existing `Signable` send path, using the
same API-key auth. Non-404 failure â†’ abort with the API's message, **no DB changes**.

### Amend action

`amendContract(Bands $band, Bookings $booking)` on the web `BookingsController` and
`Api\Mobile\BookingsController`, both delegating to a shared
`BookingService::amendContract(Bookings $booking)` method (the mobile controller
already injects `BookingService`; guards and the transaction live there once).

- **Guards** (422 with a human-readable message): `contract_option === 'default'`;
  contract exists with status `sent`; booking status `pending`.
- **Order:** void the PandaDoc document first (external call, not rollback-able),
  then in a DB transaction: contract â†’ status `pending`, `envelope_id` â†’ null;
  booking â†’ status `draft`. `custom_terms`, `buyer_name_override`, `asset_url` are
  untouched (resend regenerates the PDF and creates a fresh document, as today).
- **Routes:** `POST /api/mobile/bands/{band}/bookings/{booking}/contract/amend`
  inside the `mobile.band:write:bookings` group, returning the formatted booking
  (same shape as `sendContract`); matching web POST route redirecting back.

### contract_option on mobile update

`UpdateBookingRequest` gains `contract_option => sometimes|in:default,none,external`,
rejected while the booking's contract status is `sent` or `completed` (that state
belongs to amend / stays locked). No automatic booking-status side effects â€” parity
with web's update semantics; the create-time captions describe create-time behavior.

## Error handling

- Wrong-state amend â†’ 422 surfaced as an alert in both UIs.
- PandaDoc void failure (non-404) â†’ error alert, nothing mutated.
- Amend after the doc was hand-deleted in PandaDoc â†’ succeeds (404-tolerant).
